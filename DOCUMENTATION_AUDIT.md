# Documentation Audit — Correctness and Completeness

**Date:** 2026-07-21  
**Code baseline:** `main` after #1207 and #1237, plus the XML and JUnit scope corrections in this QA change  
**Scope:** Root documentation, module documentation, build/version claims, cleanup capability descriptions, CLI/product/release claims, and QA records.

## Objective

This audit checks whether documentation describes the software that actually exists. It does not treat a planned feature, disabled test, build artifact, or open issue as completed functionality.

## Verification method

Documentation was compared with:

- root and module Maven/Tycho configuration;
- Eclipse target/product/update-site definitions;
- cleanup constants, extension registrations, and implementation classes;
- test inventory generated by CI;
- current CLI transaction boundaries;
- merged and closed pull-request state;
- the planned multi-file cleanup implementation and JDT UI fork patch;
- XML cleanup transformer, stylesheet, preference wiring, and tests;
- cleanup-scope expansion behavior for local and coordinated options.

## Corrections verified

### Root build and release documentation

#1237 synchronized current Tycho references in `README.md`, `CONTRIBUTING.md`, `pom.xml`, and Copilot build references to 5.0.3. It also removed the contradictory claim that versioned releases were suitable for production while all plug-ins were described as experimental.

The README now distinguishes:

- versioned experimental releases;
- development snapshots;
- the current per-file CLI behavior;
- the implemented versus planned JUnit and Int-to-Enum scope.

### Planned multi-file cleanups

The documentation now matches the implementation merged through #1207 and the JDT UI fork patch merged through #94:

- one immutable semantic plan per project and cleanup run;
- file-local AST rewrites combined by the existing LTK cleanup pipeline;
- no retained planning AST nodes;
- optional target-scope expansion through the patched JDT UI bundle;
- save actions remain single-file;
- the current headless application is not yet a project-wide atomic transaction.

The previous statement that Int-to-Enum was merely a placeholder is no longer correct. The implemented automatic scope remains deliberately narrow and rejects public/protected APIs, arbitrary integer propagation, aliases, flags, unresolved uses, and external numeric semantics.

This QA change also corrects JUnit scope gating: local-only JUnit options no longer request every project source unit. Full-project expansion remains enabled only for the currently coordinated `RULEEXTERNALRESOURCE` migration. The single Int-to-Enum option still combines local and project-wide behavior, so its minimal-scope design remains tracked in #1228 and #1212.

### Pull-request state

All open pull requests in `carstenartur/sandbox` and `carstenartur/eclipse.jdt.ui` were reviewed. Valid completed work was merged; duplicate, superseded, stale, or semantically unsafe branches were closed with technical explanations. The detailed decision record is in [`docs/qa/open-pr-review-2026-07-21.md`](docs/qa/open-pr-review-2026-07-21.md).

Both repositories had zero open pull requests at the end of that review.

### XML cleanup

The previous audit correctly identified that `XML_CLEANUP_INDENT` was wired through the cleanup layers but ignored by `SchemaTransformationUtils`. That mismatch is corrected in this QA change:

- compact mode sets `OutputKeys.INDENT=no`;
- indented mode sets `OutputKeys.INDENT=yes`;
- the stylesheet strips formatting-only whitespace before serialization;
- tests require a visible layout difference, semantic equality, and compact default behavior.

`ARCHITECTURE.md` and `TODO.md` were rewritten to describe the actual JDT and standalone PDE execution paths, security properties, tests, and remaining limitations.

### Documentation inventory

Earlier audit statements about missing README files are obsolete: the current inventory records README coverage for the mining CLI, JGit modules, Maven plug-in, AST/JDT bridge, and packaging modules. Remaining architecture/TODO gaps are listed explicitly rather than being described as missing user documentation.

## Remaining verified gaps

Before advertising general automatic multi-file cleanup support:

1. split or candidate-gate Int-to-Enum local versus project-wide behavior (#1228, #1212); the JUnit sub-option defect is fixed in this QA change;
2. prove atomic apply/undo, stale-plan rejection, save-action isolation, and multi-project lifecycle behavior (#1213);
3. reconcile/build the transformed source set in positive integration tests (#1222);
4. replace remaining unconditional full-project discovery with candidate-driven JDT search (#1212).

### P1 — scale and user transparency

### Compatibility and broader transformations

- enum numeric/public compatibility policy (#1216);

- generated-name collision policy (#1225);

### Headless execution

`sandbox_cleanup_application` still creates a cleanup refactoring per file. Atomic project grouping, complete rollback, deterministic diff/report behavior, and multi-file planning are tracked in #1210.

### Product and release evidence

- reproducible patched JDT UI bundle/product/update-site path (#1209);
- install, resolution, startup, and cleanup smoke tests for published artifacts (#1215);
- exact post-merge commit-to-artifact/test provenance (#1218).

### XML cleanup

The indentation preference is fixed, but broader production-readiness still requires:

- resource-charset policy instead of unconditional UTF-8 decoding/writing;
- dirty-editor and external modification integration tests;
- malformed-input/user-status tests;
- PDE/schema validation after transformation;
- scale measurements for large files and project batches.

## Documentation maintenance rules

A capability or version change is complete only when:

1. source/build metadata is updated;
2. module README/architecture/TODO status is updated;
3. root capability wording is checked;
4. positive and negative tests identify the supported boundary;
5. known limitations link to an issue or are stated in the module roadmap;
6. release/product claims are no stronger than their automated gates.

Long term, #1219 should replace manually duplicated version/capability tables with one validated inventory.

## Conclusion

The documentation now represents a coherent experimental project rather than a collection of independent historical claims. The most important remaining work is explicitly separated into cleanup correctness, scale, headless transactions, compatibility, and product/release evidence. No current document should describe the project or its published artifacts as generally production-ready.
